# Preparing the environment

Our laboratory environment requires several elements:
- A resource group, which acts as a directory within Azure where we will generate all the instances of this project.
![imagen](https://github.com/user-attachments/assets/1d712cdc-e4f6-4282-bff5-84bb5f7527d3)
- In order to have a virtual machine that acts as our honeypot, we must first enable a VNet, which will assign an IP to the machine.
![imagen](https://github.com/user-attachments/assets/4877f4a8-779b-4614-8203-28866dd733f9)
- The virtual machine, which will be the honeypot that attackers will try to access, and as a result will be our way of obtaining data from real attacks.

Of these three elements, it is particularly important to configure the virtual machine we are going to use correctly, as this significantly affects the final cost of the lab and the type of events we will be working with. In my case, I opted for a Windows 10 x64 gen.2 machine and configured it to be accessed via the RDP service using port 3389. It should also be noted that this laboratory does not require a premium hard drive; a standard HDD will suffice, as we will not be using the machine itself.

You will see that we have more items than those mentioned in our resource group. This is because Azure generates several items derived from the configurations we previously established. Below is a guide to what the new items are:

- m1-ss-t6651_z1: eth port; network interface
- M1-SS-T6-nsg: firewall
- M1-SS-T6_OsDisk_1_8c955c2a03c644fb839c6b3b9acc1998: disk
- M1-SS-T6-ip: VM public IP
